# tools/net — EDGE realtime networking host tools

## `edge_realtime_peer.py`

Generic stdlib-only UDP peer for the realtime networking diagnostic demo
(`demo/edge_net_realtime_meter.cpp`, EDGE Stage 9S.3).

The transport is deliberately generic — it moves fixed-size UDP datagrams and
knows nothing about Atari, SIO, POKEY, FujiNet, or Netstream. The only
EDGE-specific knowledge (the `MeterPacket16` field layout) is isolated in the
`meterpacket_*` helpers and is used only to fill the echo/ticker timing fields
and, optionally, to pretty-print packets.

### Usage

```sh
# Echo (default): reply once per received 16-byte packet, stamping T2/T3/peer_seq.
python3 tools/net/edge_realtime_peer.py --mode echo --tv ntsc --decode

# Ticker, explicit target (RX-only client test):
python3 tools/net/edge_realtime_peer.py --mode ticker --target-host 192.168.1.42 --target-port 6502

# Ticker, learn the client's address from its first packet:
python3 tools/net/edge_realtime_peer.py --mode ticker
```

### Options

| Option | Default | Meaning |
|--------|---------|---------|
| `--host` | `0.0.0.0` | bind address |
| `--port` | `9000` | UDP port |
| `--mode echo\|ticker` | `echo` | echo replies per packet; ticker streams packets |
| `--size` | `16` | expected packet size; non-16-byte datagrams are ignored |
| `--tv ntsc\|pal` | `ntsc` | TV standard selecting the virtual jiffy rate |
| `--hz <float>` | derived | override the virtual jiffy rate |
| `--ticker-hz <float>` | `30` | ticker send rate (packets/sec) |
| `--target-host` / `--target-port` | learn | ticker explicit destination (must be given together) |
| `--decode` | off | pretty-print decoded `MeterPacket16` fields |
| `--quiet` | off | suppress per-packet logs (keep the once-per-second summary) |

### Byte-stream reassembly (important)

Netstream (UDP-seq) carries an **unframed byte stream**. The FujiNet firmware forwards
serial bytes to UDP in whatever chunks they arrive, so a 16-byte MeterPacket can split
across two datagrams (or several can share one). The peer therefore **reassembles**
16-byte units from the incoming byte stream (`Reassembler`) instead of assuming one
datagram == one packet — the same thing the Atari EDGE adapter does on its serial RX
ring. Without this, intact packets that happened to split across datagrams are miscounted
as `bad`/lost even though the data is perfect. If a datagram is lost/reordered the stream
desyncs (no framing/checksum to recover); the peer resyncs on the `E7 01` magic+version
marker and reports the discarded bytes as `resync NB`.

### Reliability + throughput readouts

Both ends turn the raw counters into measured rates and loss. The numbers are
diagnostic estimates for the lossy raw 16-byte Netstream path (no retransmit).

**Peer (authoritative forward path).** The peer sees every `client_seq`, so each
report line (`--stats-interval`, default 1 s) measures the sender→peer direction
exactly by sequence-gap analysis:

```
echo 1.0s: rx 15.0pps 240B/s  fwd_loss 0.0% (15/15)  dup 0 reorder 0 resync 0B bad 0 | tx 15.0pps 240B/s
```
- `rx … pps … B/s` — true receive throughput (packets/s and bytes/s = pps×16), measured
  after reassembly (datagram fragmentation does not affect it).
- `fwd_loss N% (got/expected)` — forward loss from the `client_seq` span seen in the
  window (`expected = max−min+1`, signed-wrap aware).
- `dup` / `reorder` — duplicate and out-of-order `client_seq` counts.
- `resync NB` — bytes discarded resyncing the stream after a lost/reordered datagram.
- `bad` — reassembled units whose body still failed to decode (should be ~0).
- `tx …` — reply throughput.

> **Throughput note (emulator).** In the fujinet-pc + NetSIO + Altirra stack, the
> Netstream external TX clock is generated by the (busy-waiting) firmware and paced over
> NetSIO, so each 16-byte serial transmit stalls the emulated CPU and the Atari frame loop
> drops to ~12–60 fps depending on host/firmware load — capping the packet rate. This is
> an emulation-stack characteristic, **not** packet loss and not the EDGE code; with the
> reassembly fix the forward path measures 0 % loss / 0 corruption. Real FujiNet hardware
> (real POKEY at 31250 baud) would not have this stall.

On Ctrl-C the peer prints a cumulative `summary:` line (lifetime totals + overall
forward loss).

**Atari HUD (live, directional).** The `LINK QUALITY (1S)` block recomputes once per
~1 s RTCLOK window. Because the packet carries the peer's reply counter (`peer_seq` =
forward survivors), the client splits loss by direction with no packet change:
- `TXHZ` / `RXHZ` — send / receive packets per second.
- `TXBPS` / `RXBPS` — bytes per second (= Hz × 16).
- `LOSS RT` — round-trip loss (sent vs replies received).
- `FWD` — forward loss Atari→peer (`1 − Δpeer_seq/Δtx`); should track the peer's
  authoritative `fwd_loss`.
- `REV` — reverse loss peer→Atari (`1 − Δgot/Δpeer_seq`).
- `WORST` / `BEST` — round-trip loss range seen so far.

The peer's `fwd_loss` is the ground truth for the forward direction; the HUD's `FWD`
is the client-side estimate of the same thing, and `RT ≳ FWD` and `RT ≳ REV`.

### Time model

The peer keeps a virtual jiffy clock in Atari-compatible `u16` units (wraps mod
65536), advanced by host monotonic time and seeded from the first valid client
packet's `T1` (coarse — one-way latency is included). Default jiffy rates are
derived from the standard ANTIC frame geometry, `cpu_frequency / (scanlines *
114)`:

- NTSC: `1789773 / 29868 = 59.9227 Hz`
- PAL:  `1773447 / 35568 = 49.8607 Hz`

`--hz` overrides this. These diagnostics are demo-grade, not a production
clock-sync protocol.

### Demo client configuration

The Atari `.xex` has no command line. Set the peer endpoint by editing the
top-of-file constants in `demo/edge_net_realtime_meter.cpp`:

```cpp
static constexpr const char* kPeerHost  = "192.168.1.50";  // host running the peer
static constexpr u16         kPeerPort  = 9000;
static constexpr u16         kLocalPort = 0;
```

`MeterPacket16` is a demo diagnostic payload only — **not** an EDGE wire format.

### End-to-end Mode B test (fujinet-pc + NetSIO + Altirra + Docker peer)

The realtime lane needs the EDGE Netstream adapter — build the demo with
`EDGE_ATARI_FUJINET_REALTIME_NETSTREAM=ON` (the CMake target then links the two
`.S` handlers). Without it the realtime HAL is the stub: the HUD shows `ACTIVE` /
`LAST OK` but no real SIO happens and this peer receives nothing.

With the NetSIO hub + fujinet-pc firmware already running (Altirra's `netsio`
custom device connects to the hub), and the peer reachable on a **distinct IP** so
the firmware's `0.0.0.0:port` bind + send-to-host does not self-loop:

```sh
# 1. Peer in Docker at a fixed IP (the firmware sends here).
docker network create --subnet 172.30.0.0/24 nspeer
docker run -d --name nspeer-rt --network nspeer --ip 172.30.0.2 \
    -v "$PWD/tools/net:/peer:ro" python:3-slim \
    python -u /peer/edge_realtime_peer.py --mode echo --port 9000 --tv ntsc --decode

# 2. Build the demo with the real adapter, pointed at the peer.
cmake -S . -B build-ns -DEDGE_ATARI_FUJINET_REALTIME_NETSTREAM=ON \
      -DEDGE_NET_PEER_HOST=172.30.0.2 -DEDGE_NET_PEER_PORT=9000
cmake --build build-ns --target edge_net_realtime_meter

# 3. Run in Altirra Mode B (keeps the netsio device); watch the peer.
scripts/altirra_probe.sh build-ns/edge_net_realtime_meter.xex B   # or launch interactively
docker logs -f nspeer-rt

# teardown
docker rm -f nspeer-rt && docker network rm nspeer
```

Validated this way 2026-06-15: HUD `ACTIVE`, `TX`/`RX`/`GOT` advancing, `DELAY ~4f
(~66ms)`, signed `OFFSET`, `RTT`, `JIT`, with realistic loss on the raw 16-byte
Netstream path (no retransmit). **Not** validated on physical FujiNet hardware.
